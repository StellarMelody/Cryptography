二、思考题完整参考答案
1. 数字签名的三大保证
数字签名提供完整性、身份认证、不可否认性三大保证：
完整性
含义：消息在传输、存储过程中没有被篡改、增删。
实验实现：先对密文计算 SHA256 哈希，再对哈希签名；验证时重新计算密文哈希与签名内哈希对比，只要密文改动，哈希立刻变化，验证失败，发现篡改。
身份认证（真实性）
含义：能确认消息确实由对应发送方发出，不是第三方伪造。
实验实现：签名仅能用发送方独有的签名私钥生成；任何人只能用配套公钥验证，第三方没有私钥无法伪造合法签名，确认发送者是持有该私钥的张三。
不可否认性
含义：发送方无法否认自己发送过这条消息、生成过该签名。
实验实现：私钥仅发送方持有，只有他能产生合法签名；第三方可独立用公钥核验，发送方无法抵赖。
2. 为什么要 “先哈希后签名”
RSA 明文长度有限制：2048 位 RSA 仅能直接加密 / 签名小于 245 字节数据。本次原始消息虽短，但密文是 256 字节，超过直接签名上限，无法直接签名；哈希后固定输出 32 字节短摘要，适配 RSA 输入限制。
提升签名效率：哈希算法计算速度远快于大数模幂 RSA 运算。大文件若直接 RSA 签名计算极慢；先哈希把任意长度数据压缩为固定短摘要，大幅减少签名运算量。
安全性更强：直接对原文 RSA 签名存在可延展性攻击；对哈希摘要签名可规避这类漏洞，同时哈希的抗碰撞特性保障篡改可被检测。
统一固定输出长度：无论几 KB、几 GB 文件，SHA256 永远输出 32 字节摘要，签名长度统一，方便标准化处理。
3. 签名长度与密钥长度的关系
2048 位 RSA 签名为 256 字节原因：
RSA 密钥长度单位是比特 (bit)，1 字节 = 8 比特。2048 bit ÷ 8 = 256 Byte。RSA 签名输出是模 N 的大数，数值范围 0~N-1，N 是 2048 位模数，因此签名二进制固定占用 2048 比特，即 256 字节。
4096 位 RSA 签名长度：4096 ÷ 8 = 512 字节。
4. 哈希函数三大安全性质
（1）抗原像性（单向性）
定义：给定哈希输出 h，无法在计算可行时间内找到任意原文 m，满足 Hash (m)=h。
风险：不满足则攻击者拿到哈希值就能反推出原始消息，签名失去保密与认证意义。
（2）抗第二原像性（弱抗碰撞）
定义：给定原文 m1，无法找到另一条不同消息 m2，使得 Hash (m1)=Hash (m2)。
风险：攻击者拿到合法消息 + 签名，能构造另一条不同内容但哈希相同的消息，替换原文后签名依然验证通过，篡改内容不被发现。
（3）抗碰撞性（强抗碰撞）
定义：无法找到两条完全不同的消息 m1、m2，使得 Hash (m1)=Hash (m2)。
风险：攻击者提前构造一对哈希相同的文件，先用 m1 获取合法签名，再替换为恶意 m2，签名核验依旧通过，伪造任意文件的有效签名。
5. SHA-1 的安全问题
淘汰原因：SHA-1 存在公开碰撞攻击，不再满足抗碰撞安全要求，无法用于数字签名、证书等高安全场景；NIST 早已明令淘汰 SHA-1，强制使用 SHA256 及以上。
2017 谷歌碰撞攻击影响：研究者构造出两段不同文件，SHA-1 哈希完全一致。针对签名的致命漏洞：攻击者先拿无害文件向机构申请合法数字签名，随后把文件替换为恶意木马 / 伪造文书，哈希不变，原有签名依然校验成功。
举例：攻击者制作两份 PDF，一份正常合同、一份伪造欠款合同，SHA1 哈希相同。先用正常合同获取企业合法签名，发送伪造欠款合同时附带原有签名，接收方校验哈希与签名均合法，无法识别文件被替换。
6. 公钥与私钥的角色
（1）签名密钥对
签名私钥 signature_private_key.pem：发送方持有，生成数字签名；
签名公钥 signature_public_key.pem：对外公开，接收方验证签名合法性。
（2）加密密钥对
加密公钥 encryption_public_key.pem：对外公开，加密明文，只有对应私钥能解密；
加密私钥 encryption_private_key.pem：接收方持有，解密密文。
（3）攻击者获取公钥后的能力
能做：加密消息发给私钥持有者、验证他人发布的数字签名；
不能做：解密任何密文、伪造合法数字签名、推导对应私钥。
7. 签名与加密的本质区别
核心作用区分
消息加密（RSA 公钥加密）
解决消息保密性：防止第三方窃听读取明文；使用接收方公钥加密，只有接收方私钥可解密。
不提供身份认证、不可否认性，任何人持有公钥都能加密，无法判断发送者。
数字签名（RSA 私钥签名）
解决完整性、身份认证、不可否认：证明消息未篡改、发送者身份、发送方无法抵赖；使用发送方私钥签名，所有人用公钥核验。
不提供保密性，签名本身可公开，任何人都能查看摘要与签名。
为什么加密不能替代签名
加密仅保障机密，无法证明发送者身份：攻击者截获密文删除后，自己用接收方公钥伪造一段加密消息，接收方无法区分是谁发送；加密也无法提供不可否认性。而签名依靠发送方独有私钥，唯一标识发送人，二者目标完全不同，不能互换。
本实验流程逻辑
先加密保证消息不泄露，再对密文签名保证密文完整、来源可信；接收方先验签确认密文没被篡改、来源合法，再解密读取原文。
8. 不可否认性与 MAC 的区别
不可否认性定义：消息发送者无法事后否认自己生成、发送过该消息；第三方权威机构可独立核验证据判定责任。
数字签名支持不可否认：签名基于非对称密码，公私钥分离，签名私钥仅发送方独占，公钥全网公开可供所有人核验。发生纠纷时，法官 / 第三方可使用公钥独立验证签名，确认只有发送方能生成该签名，发送方无法抵赖。
MAC（消息认证码）无法提供：MAC 基于对称密钥，通信双方共用同一密钥。一旦出现纠纷，无法区分是发送方、接收方哪一方生成的 MAC 值，双方都可以伪造，没有可信第三方核验依据，因此做不到不可否认。
9. 数字签名现实应用场景
软件安装包 / 固件代码签名
场景：Windows 驱动、手机系统固件、开源软件安装包。
解决问题：防止黑客植入病毒篡改安装包；用户下载后校验开发商签名，确认程序未被劫持、来源官方。
电子合同、电子政务文书
场景：线上劳动合同、行政审批文件、电子发票。
解决问题：确认文书签署人真实身份，签署后内容不可篡改；产生法律层面不可否认性，具备司法举证效力。
电子邮件 S/MIME 签名
场景：企业商务邮件、涉密公文邮件。
解决问题：防止钓鱼邮件伪造发件人、拦截中途篡改邮件内容；收件人可验证邮件确实由发件人本人发出。
补充可选场景：区块链交易签名、CA 数字证书、PDF 电子签章。
10. 签名时间证明与可信时间戳 TSA
数字签名本身无法证明签名真实时间：签名仅绑定文件与私钥，本地设备系统时间可被随意篡改，攻击者可修改电脑时钟伪造签名日期，签名中不含权威可信时间凭证。
验证真实时间方案：可信时间戳服务 TSA
流程：用户对文件 / 签名哈希，发送至权威第三方时间戳服务器（TSA）；TSA 使用自身私钥，绑定国家标准可信时钟时间，对哈希生成时间戳签名。
作用：时间戳由中立权威机构背书，时间不可篡改、不可伪造。一年后核验时，同时校验文件签名 + TSA 时间戳，即可确认签名在 TSA 标注的真实时间生成，当事人无法篡改、谎称签署时间。
11. RSA 密钥长度安全与性能权衡
1024 位 RSA：不安全。算力提升 + 大数分解算法优化，学术机构已能低成本破解 1024 位模数；各国安全标准（国密、NIST）2010 年后全面禁止 1024 位用于签名、证书，仅临时低安全加密可短期使用，长期存在破解风险。
2048 位 RSA（本实验选用）：安全与性能平衡最优。短期（10–20 年）无公开可行破解手段，符合国内、国际商用密码标准；模幂运算速度快，签名 / 加密耗时短，硬件、软件兼容性最好，是当前通用标准密钥长度。
4096 位 RSA：安全强度更高，抗量子前计算攻击能力更强，但性能损耗极大。密钥、密文、签名体积翻倍，加解密、签名核验运算量提升数倍，占用更多 CPU / 存储；仅用于最高安全等级场景（根 CA 证书、国家涉密系统），普通业务、教学实验无需使用，过度牺牲性能。
12. 多层签名（签名的签名）与多重签名
Alice 签名后 Bob 再对 Alice 的签名签名，存在意义：有意义，属于嵌套签名 / 层级签名，用于信任链传递。
Alice 对原始文件签名，证明文件内容可信；Bob 对 Alice 的签名二次签名，代表 Bob 认可 Alice 这份文件与签名，形成二级信任背书。
适用场景：
（1）多级审批流程：员工 Alice 起草文件签名，部门主管 Bob 复核后二次签名，代表两级均认可文件；审计时可逐层核验每一级签名，追溯审批链路。
（2）证书信任链：根 CA 签发二级 CA 证书（根签名二级公钥），二级 CA 签发用户证书（二级 CA 签名用户公钥），层层签名构建可信证书链。
多重签名（多人共同签名一份文件）：另一种场景，一份文件需要 N 个人中至少 K 人签名才生效（门限多重签名），例如企业大额转账需要财务、法人、风控三方同时签名才能发起交易。
